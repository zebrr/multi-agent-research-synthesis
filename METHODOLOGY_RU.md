# Методология комплексных исследований с мультиагентным подходом

Используйте фазы как проектные или агентские инструкции для вашей модели или CLI-агента, адаптируйте под свои потребности.

---

## Фаза 1: Подготовка задания на research

### Введение

На вход поступает несистематизированный запрос от заказчика (команда, стейкхолдер, собственная гипотеза). В режиме диалога с SOTA-моделью (ChatGPT, Claude, Gemini, ...) запрос трансформируется в структурированное задание на Deep Research.

Задание проектируется итеративно: уточняем scope, формулируем критерии, согласовываем теги и шаблоны. Готовое задание должно быть самодостаточным — research-агент работает по нему без дополнительных пояснений.

Структура ниже может использоваться как часть проектной инструкции для систематического проектирования исследований. Конкретное задание строится по этим критериям, адаптируясь под домен и задачу.

### 1.1. Контекст и цель

**Роль агента** — 1-2 предложения, фокус и экспертиза.

**Цель исследования** — что исследуем и зачем. Это ключевой раздел, который определяет всё остальное: критерии включения/исключения, выбор тегов, фокус карточек. Формулировка цели должна давать агенту чёткое понимание, какие кейсы релевантны, а какие — мусор.

**Ограничения:**
- География (фокус + что включать с осторожностью)
- Временной период (год запуска/обновления)
- Язык источников и итогового отчёта

### 1.2. Scope

**Что Включаем** — типы объектов исследования с примерами. Конкретика помогает агенту калибровать границы.

**Что НЕ включаем** — явные исключения с примерами. Без этого агент притащит смежный мусор.

**Исключения из исключений** — когда объект из "не включаем" всё-таки включаем (например: "ChatGPT — только если есть институциональное внедрение с измерением результатов").

### 1.3. Eligibility (критерии отбора кейса)

Минимальные требования — все должны выполняться одновременно:
- Соответствие scope
- Временной период
- Масштаб (не единичный эксперимент)
- Источники: ≥2, из них ≥1 P1

**Negative cases (paused/discontinued)** — нужно включать если:
- Остальные критерии соблюдены
- Публично описана причина остановки

### 1.4. Теги и шкалы

Оформлять таблицами — читаемость выше, агент точнее следует.

**Тип решения** — domain-specific, 5-8 тегов максимум. Больше = агент путается.

**Другие классификаторы** — по необходимости домена (аудитория, модель внедрения, модель оплаты, технологии и т.п.).

**R0-R3 (зрелость результатов)** — универсальная шкала:

| Тег | Описание |
|-----|----------|
| R0 | Только анонс, нет данных о ходе/результатах |
| R1 | Качественные данные: отзывы, описания эффектов словами |
| R2 | Количественные данные без строгой методологии (нет контрольной группы, self-reported) |
| R3 | Строгие данные: контрольная группа, longitudinal, peer-reviewed, или очень сильная публичная доказательная база |

**Статус** — опционально, но показывает себя как универсальный:

| Тег | Описание |
|-----|----------|
| ACT | Active — работает |
| PAU | Paused — приостановлено |
| DIS | Discontinued — закрыто |

Теги и шкалы необходимо зафиксировать разделом Legend в задании на исследование.

### 1.5. Политика источников

**Категории:**

| Тег | Что включает |
|-----|--------------|
| P1 | Официальные страницы, пресс-релизы, case studies, peer-reviewed |
| P2 | Медиа, интервью, обзоры, блоги экспертов |
| P3 | Посты без подтверждений, форумы, отзывы |

**Требования:**
- Минимум на кейс: ≥2 источника, из них ≥1 P1
- URL обязателен (полный, без сокращений до домена)
- Проверка URL: пробовать fetch → если не работает, то search по URL → если не получается, добавить метку `[unverified]`
- `[unverified]` — допустимо, источник не отбрасывается, будет проверен отдельно на этапе нормализации

### 1.6. Выходной формат

**Формат документа: Markdown (.md)**

**Структура отчёта:**
1. **Legend** — все теги и шкалы с расшифровками (копия из задания)
2. **Methodology** — как искал, какие запросы, ограничения
3. **Casebook** — кейсы C01...Cxx по шаблону
4. **Observed Patterns** — паттерны со ссылками на [Cxx] и [Sxx], без выводов и рекомендаций
5. **Appendix — Sources** — список источников в формате [Sxx]

### 1.7. Шаблон карточки (Casebook)

Каждый кейс получает уникальный ID: **C01, C02, ... Cxx**

```markdown
### [Cxx] — [Название]: [Краткое описание]

**Базовая информация** // Обязательно по критериям исследования
- Название: ...
- Регион: ...
- Год запуска: ...
- Статус: ACT/PAU/DIS — [расшифровка]

**Классификация**  // Обязательно по критериям исследования
- Тип решения: [TAG] — [расшифровка словами]
- [Другие теги]: [TAG] — [расшифровка]
- Зрелость результатов: [R0-R3] — [расшифровка]

**Что это / как работает** // Набор полей по необходимости
[2-3 абзаца]

**Результаты и метрики** // Набор полей по необходимости
[1-2 абзаца]

**Источники** // Обязательно!
[S001], [S002], ...
```

Все теги сопровождаются словесной расшифровкой прямо в карточке.

### 1.8. Observed Patterns

Паттерны, которые агент заметил "по горячим следам" после обработки всех кейсов.

**Требования:**
- Каждый паттерн подкреплён ссылками на кейсы [Cxx] и/или источники [Sxxx]
- Только наблюдения, никаких выводов и рекомендаций
- `Inference` (собственные выводы агента) помечать явно

### 1.9. Appendix — Sources

Список всех источников:

```
[S001]: P1, https://full.url/path/to/page, "Official Case Study Title"
[S002]: P2, https://another.url/article
...
[S103]: P1 [unverified], https://blocked.site/report, "Annual Report 2024"
```

Название опционально, но если есть — в кавычках после URL.

### 1.10. Контроль качества

Чек-лист перед выдачей:
- [ ] Все теги соответствуют Legend
- [ ] Все кейсы имеют C-ID формата [Cxx]
- [ ] Все факты/метрики имеют [Sxxx]
- [ ] Все [Sxxx] из текста есть в Appendix в заданном формате
- [ ] `Inference` помечены явно
- [ ] URL полные (не сокращены до домена)

## Фаза 2: Мультиагентный сбор

- Одно задание → загружается в несколько (3+) разных агентов доступным способом (приложение, API)
- Основные: OpenAI (Agent и DeepResearch — два разных!), Claude Research, Gemini DeepResearch
- Опционально: Perplexity Labs, Qwen DeepResearch, Kimi Researcher, Manus Wide Research, ...
- Каждый работает независимо
- Результаты: ~50-200KB на агента, несколько документов
- Качество и структура варьируются: агенты могут игнорировать части задания, использовать свои форматы тегов/ссылок, добавлять незапрошенные секции — это нормально и учтено в методологии

## Фаза 3: Подготовка и структурная нормализация

**Цель:** привести сырые отчёты агентов к единой структуре для последующей работы со ссылками и синтеза.

- Файлы большие (50-200KB) → работать через bash/grep, не читать целиком
- Строго один файл за раз, сброс контекста между файлами
- Задание (TASK.md) — эталон: сверять структуру и теги каждого файла с требованиями задания

### Кодирование агентов

Для именования WRK-файлов и маппинга использовать стандартные сокращения:

| Код | WRK-файл |
|-----|----------|
| Claude | WRK_Claude_Research.md |
| Gemini | WRK_Gemini_DeepResearch.md |
| OAI_DR | WRK_OpenAI_DeepResearch.md |
| OAI_Ag | WRK_OpenAI_Agent.md |
| Perplx | WRK_Perplexity_Labs.md |
| Qwen | WRK_Qwen_Research.md |
| Kimi | WRK_Kimi_Researcher.md |
| Manus | WRK_Manus_Wide_Research.md |

Если в исследовании используется агент не из списка — добавить в отчёт с аналогичным форматом.

### 3.1. Создание рабочих копий

- Создать папку `_work` в директории исследования
- Скопировать все отчёты агентов (кроме задания) как `WRK_<original_name>.md`
- Первичные файлы остаются нетронутыми (READONLY на всех последующих этапах)
- Создать `NORMALIZATION_REPORT.md` с шаблоном (см. 3.5)

### 3.2. Первичный аудит

Для каждого файла — оценка состояния:

| Критерий | Что смотрим |
|----------|-------------|
| Структура | Есть ли секции из задания? В каком порядке? |
| Карточки | Есть ли Casebook? Сколько кейсов? |
| Теги | Коды или описания? Какой формат? |
| Ссылки | Appendix, inline, footnotes? Сколько уникальных? |
| Язык | Совпадает с заданием? |

**Результат:** заполненная сводная таблица в NORMALIZATION_REPORT.md с качественной оценкой каждого файла.

**Качественная оценка** — вывод по каждому файлу:
- ✅ Готов к нормализации — структура близка к заданию
- ⚠️ Требует значительной работы — структура далека, но данные ценные
- ❌ Рекомендуется использовать только как источник кейсов — формат несовместим

Финальное решение по файлам с оценкой ⚠️ и ❌ принимает пользователь.

### 3.3. Структурная нормализация

Для каждого файла (по одному, последовательно):

**Чек-лист:**

- [ ] Секции соответствуют "Выходному формату" из задания
- [ ] Перепутанные секции → переставить
- [ ] Пропущенные секции → создать с меткой `<!-- MISSING: Данные не предоставлены агентом -->`
- [ ] Уникальные секции (которых нет в задании) → вынести в конец с меткой `<!-- UNIQUE: использовать на этапе Cross-synthesis -->`
- [ ] Нет Appendix со ссылками (inline-формат) → пометить `<!-- MISSING: Appendix Sources — ссылки inline, восстановить на Фазе 4 -->`
- [ ] Теги в карточках: где описания вместо кодов → добавить коды, где невозможно определить → `[TAG:UNKNOWN]`
- [ ] Формат тегов единообразен внутри файла (чтобы корректно обрабатывались при синтезе)
- [ ] Легенда (Legend) удалена из файла (если была)

**После обработки файла:**
- Обновить детали в NORMALIZATION_REPORT.md
- Сбросить контекст
- Перейти к следующему файлу

### 3.4. Проверка качества

После обработки всех файлов — финальная проверка:

- [ ] Все файлы из сводной таблицы обработаны
- [ ] Структура секций соответствует заданию (или помечена MISSING/UNIQUE)
- [ ] Теги в формате кодов (или помечены TAG:UNKNOWN)
- [ ] Легенды удалены из всех файлов
- [ ] NORMALIZATION_REPORT.md актуален

### 3.5. Выходные артефакты

**Папка `_work`:**
- `WRK_*.md` — структурно нормализованные файлы
- `NORMALIZATION_REPORT.md` — отчёт (создаётся при обработке первого WRK-файла, обновляется для остальных)

**Шаблон NORMALIZATION_REPORT.md:**

```markdown
# Отчёт о нормализации: [Название исследования]

## Статистика по структуре (фаза 3)

| Файл | Оценка | Карточек | Ссылок | Проблемы |
|------|--------|----------|--------|----------|
| WRK_*.md | ✅/⚠️/❌ | N | N | MISSING: ..., UNIQUE: ... |

## Детали по файлам

### WRK_[Agent].md
- **Оценка:** ✅/⚠️/❌ — обоснование
- **Структура:** соответствует / исправлено (что именно)
- **Теги:** формат, статус нормализации
- **Ссылки:** формат (Appendix / inline / footnotes), количество
- **Уникальные секции:** список или "нет"
- **Язык:** RU / EN
- **Примечания:** особенности файла
```

## Фаза 4: Нормализация ссылок

**Цель:** привести ссылки в каждом WRK-файле к единому формату, проверить что все ссылки используются и все утверждения атрибутированы.

**Принцип:** один WRK-файл за раз, изолированно, сброс контекста между файлами (как в Фазе 3).

### 4.1. Инвентаризация

Извлечь два множества:

**T (текст)** — все уникальные URL из текста файла:
- inline markdown `[name](url)`
- footnotes `[^1]: url`
- голые URL
- ссылки на существующий Appendix `[Sxxx]` → достать URL по ID

**A (Appendix)** — все URL из Appendix (если есть):
- Если MISSING (помечен на Фазе 3) → A = ∅

**Результат:** два списка URL для сравнения.

### 4.2. Нормализация Appendix

Сравнить T и A:

| Случай | Условие | Действие |
|--------|---------|----------|
| **MISSING** | A = ∅ (Appendix не был создан агентом) | Создать Appendix из T |
| **Неполный** | T ⊄ A (есть в тексте, нет в Appendix) | Добавить недостающие записи |
| **Избыточный** | A ⊄ T | Обработать на шаге 4.3 |
| **Полный** | T = A | Ничего не делать |

**Формат записей:** `[Sxxx]: Px, URL, "name"` (name опционально)

**Присвоение S-ID:** сквозная нумерация S001, S002, ... для новых записей. Если записи уже есть — продолжать нумерацию.

### 4.3. Конвертация и проверка unused

**Конвертация:**
- Заменить все inline/footnotes/голые URL в тексте на `[Sxxx]`
- Очистить URL от мусорных якорей (`#:~:text=...`, `#section`), но НЕ сокращать до домена

**Обработка дублей:**
- Если после очистки якорей два URL стали одинаковыми → оставить один S-ID
- Второй S-ID удалить из Appendix
- В тексте оба места получают одинаковый S-ID
- Нумерацию НЕ перестраивать (дырки типа S001, S003, S004 — ок, поправится на этапе мэпинга)

**Проверка unused:**
- Извлечь множество S-ID из текста (используемые)
- Сравнить с множеством S-ID в Appendix
- Разница = `[unused]` → пометить в Appendix

**Контроль качества:**
- [ ] Нет голых URL в тексте (все заменены на `[Sxxx]`)
- [ ] Все `[Sxxx]` из текста существуют в Appendix (нет висячих)
- [ ] Нет дублей по URL в Appendix (один URL = один S-ID)

### 4.4. Верификация источников

**Разрешение [unused]:**
- Удалить из Appendix (источник не используется в тексте) и обновить NORMALIZATION_REPORT.md

**Разрешение [unverified] (если такие отметки есть):**
1. `web_fetch` URL
2. Если заблокирован → `web_search` по URL + ключевые слова из контекста (возможно, на английском)
3. Удалось проверить → убрать метку
4. Не удалось → оставить `[unverified]`, обновить NORMALIZATION_REPORT.md

### 4.5. Проверка атрибуции

Проверить: есть ли в тексте утверждения с цифрами/метриками без `[Sxxx]`.

Поиск семантический: по ключевым словам (effect size, %, год, название, ...), чтение диапазона, анализ.

Если найдены дырки → записать в NORMALIZATION_REPORT.md (не чинить, только фиксируем).

### 4.6. Выходные артефакты

**Чек-лист для каждого WRK-файла:**
- [ ] Appendix в формате `[Sxxx]: Px, URL, "name"`
- [ ] Нет inline URL в тексте
- [ ] Нет висячих ссылок
- [ ] Нет дублей URL в Appendix
- [ ] Нет `[unused]` (удалены)

**Файлы:**
- `WRK_*.md` — с нормализованными ссылками и Appendix
- `NORMALIZATION_REPORT.md` — обновлённый

**Дополнить NORMALIZATION_REPORT.md таблицей:**

```markdown
## Статистика по ссылкам (Фаза 4)

| Файл | Формат | Источников | Удалено (unused) | Unverified |
|------|--------|------------|------------------|------------|
| WRK_*.md | ... | N | X | Y |

### Удалённые ссылки [unused]

| Файл | S-ID | URL |
|------|------|-----|
| WRK_Agent1.md | S045 | https://... |
| WRK_Agent2.md | S012 | https://... |

### Источники [unverified]

| Файл | S-ID | URL | Используется в |
|------|------|-----|----------------|
| WRK_Agent1.md | S023 | https://... | C05, Observed Patterns |

### Утверждения без атрибуции

| Файл | Утверждение | Контекст |
|------|-------------|----------|
| WRK_Agent2.md | "g = 0.68" | Observed Patterns, строка ~120 |
```

## Фаза 5: Доатрибутирование источника (опциональная)

> ⚠️ **Опциональная фаза.** Плохо поддаётся формализации, требует проектирования под конкретный случай.

**Когда нужна:**

Если по результатам Фазы 4 для WRK-файла:
- Много удалённых `[unused]` ссылок (>30% от исходного Appendix)
- Существенная часть текста без атрибуции (много записей в "Утверждения без атрибуции")

Это ставит под сомнение ценность источника. Пользователь принимает решение:
- **Доатрибутировать** — восстановить связи между текстом и источниками (до синтеза!)
- **Отказаться** — исключить WRK-файл из синтеза

### 5.1. Генерация JSON с claims

Семантически (чтением по частям) выделить все неатрибутированные утверждения:
- Точная цитата (можно брать длинный кусок для контекста)
- Точный номер строки WRK-файла
```json
{
  "claims": [
    {
      "text": "effect size g = 0.68 for adaptive learning interventions",
      "line": 234,
      "source": null
    }
  ]
}
```

### 5.2. Проверка claims и обогащение JSON

Для каждого claim:
1. `web_search` по содержанию утверждения
2. Если найден источник → записать в JSON: URL, тип (P1/P2/P3), name
3. Если не найден → оставить `source: null`

Учитывать: claims могут быть на RU, источники в 95% на EN.
```json
{
  "claims": [
    {
      "text": "effect size g = 0.68 for adaptive learning interventions",
      "line": 234,
      "source": {
        "url": "https://...",
        "type": "P1",
        "name": "SRI Study 2020"
      }
    }
  ]
}
```

### 5.3. Обновление WRK-файла (по правилам Фазы 4)

- Использовать обогащенный JSON с claims
- Добавить найденные источники в Appendix
- Атрибутировать claims в тексте `[Sxxx]` по номерам строк из JSON

### 5.4. Обновление NORMALIZATION_REPORT.md

Добавить секцию (не трогая предыдущие):


## Фаза 6: Многопроходной синтез

### 6.1. Основные правила синтеза

**Рабочая директория:** вся работа ведётся только в папке `_work`. Оригинальные файлы агентов — READONLY на всех этапах синтеза.

**Целевое состояние:** к концу синтеза все WRK-файлы должны быть "пустыми" (только метки об использовании) — это индикатор полноты обработки.

### Тактика работы с большими файлами

Суммарный объём WRK-файлов обычно не влезает в контекст целиком. После нескольких стадий синтеза сам synthesis.md тоже становится слишком большим.

**Стратегия чтения:**
- НЕ читать WRK-файлы целиком — использовать `grep`/`search` для извлечения нужных данных
- Читать файлы частями: `head -N`, `tail -N`, `sed -n 'X,Yp'`
- Использовать `grep -A N` / `grep -B N` для контекста вокруг найденного
- При работе с карточками — извлекать по одной через grep по ID кейса

**Стратегия записи:**
- Обновлять synthesis.md инкрементально — заменять TODO-метки на контент
- После каждой стадии обновлять отчёт о выполнении
- Удалять обработанные данные из WRK-файлов (подробнее — на конкретных этапах)

**Типовые команды:**
```bash
# Найти все карточки кейсов
grep -n "^### C" _work/WRK_*.md

# Извлечь карточку по ID (с контекстом N строк)
grep -A 50 "^### C01" _work/WRK_*.md

# Найти все URL
grep -oE 'https?://[^)>\s]+' _work/WRK_*.md

# Найти UNIQUE секции (помеченные на этапе нормализации)
grep -n "<!-- UNIQUE" _work/WRK_*.md
```

### 6.3 Подготовка справочников

#### 6.3.1. Маппинг кейсов

**Цель:** построить единый справочник кейсов с дедупликацией и маппингом на оригинальные ID агентов.

**Входные данные:** карточки кейсов из всех WRK-файлов (секции Casebook).

**Действия:**
1. Извлечь все карточки из WRK-файлов (`grep -n "^### C" _work/WRK_*.md`)
2. Дедуплицировать по сущности (один продукт/организация = один кейс)
3. Присвоить финальные ID: C01, C02, ..., Cxx
4. Для каждого кейса записать:
   - `short`, `full`, `cite` — названия для использования в тексте
   - `status` — из карточки (ACT/PAU/DIS)
   - `R` — уровень зрелости результатов (R0-R3)
   - `agents` — маппинг оригинальных ID по агентам
5. Сохранить в `_work/_case_references.json` (формат см. 6.2)
   - `short` — 1–2 ключевых слова (иногда 3, если предлог/артикль)
   - `sources` — заполняется позже, на этапе синтеза карточек (6.5.1)

**Проверка:**
- [ ] Все кейсы из всех WRK-файлов учтены
- [ ] Нет дублей по сущности
- [ ] У каждого кейса есть status и R

**После завершения маппинга:** добавить ссылку на справочник кейсов в SYNTHESIS.md (секция Methodology → Справочники)

#### 6.3.2. Маппинг источников

**Цель:** построить единый справочник источников с маппингом на оригинальные ID агентов.

**Входные данные:** Appendix из всех WRK-файлов (нормализованы на Фазе 4).

**Действия:**
1. Извлечь все записи из Appendix всех WRK-файлов
2. Объединить по URL — один URL = одна запись, разные оригинальные ID в `agents`
3. Присвоить финальные ID: S001, S002, ...
4. Для каждого источника записать:
   - `url`, `type` (P1/P2/P3), `name`
   - `agents` — маппинг оригинальных ID по агентам
5. Сохранить в `_work/_sources_references.json` (формат см. 6.2)
6. Проверить маппинг выборочно (пока Appendix ещё есть, до пункта 7)
7. Удалить Appendix из WRK-файлов, добавив метку `<!-- REMOVED: Appendix → _sources_references.json -->`

**Проверка:**
- [ ] Все источники из всех Appendix учтены
- [ ] Нет дублей по URL
- [ ] Appendix в WRK-файлах заменены на метки REMOVED

**После завершения маппинга:**
- добавить ссылку на справочник источников в SYNTHESIS.md (секция Methodology → Справочники)

### 6.4. Правила атрибуции

Все ссылки в SYNTHESIS.md используют финальные ID из справочников:
- Кейсы: `_case_references.json` → C01, C02, ...
- Источники: `_sources_references.json` → S001, S002, ...

#### Форматы ссылок

| Тип | Формат | Пример |
|-----|--------|--------|
| Кейс | `Short [Cxx]` | `DreamBox [C01]` |
| Источник | `[Sxxx]` | `[S001]` |
| Несколько | `Short [Cxx], [Sxxx]` | `DreamBox [C01], [S042]` |

#### Иерархия: кейс vs источник

- Кейс — обобщённая сущность с источниками внутри
- Ссылка на `[Cxx]` покрывает все `sources` этого кейса
- Источник отдельно `[Sxxx]` — только если он НЕ входит ни в один кейс

**Не дублировать:** если утверждение подтверждается кейсом `[C01]`, не добавлять `[S001]`, если S001 ∈ C01.sources.

#### Что атрибутировать по секциям

| Секция | Атрибуция |
|--------|-----------|
| Цель, Legend, Methodology (кроме Качественного анализа), Appendix | Не требуется |
| Сравнение агентов → Качественный анализ | Кейсы и/или источники |
| Casebook (карточки) | Только источники `[Sxxx]` |
| Observed Patterns | Кейсы и источники (если не в кейсе) |
| Negative Cases | Кейсы и источники (если не в кейсе) |
| Ключевые вопросы исследования | Кейсы и источники (если не в кейсе) |
| Metrics Framework | Кейсы и источники (если не в кейсе) |
| Противоречия и верификация | Кейсы и источники (если не в кейсе) |

#### Что атрибутировать всегда

- Цифры, метрики, статистика
- Цитаты
- Конкретные факты (даты, имена, события)
- Причинно-следственные утверждения

#### Собственные выводы

Inference (выводы агента, не подкреплённые источниками) — помечать явно: `[Inference]`

#### Фиксация противоречий

Все противоречия фиксировать с атрибуцией обеих позиций:
```
По DreamBox [C01]: эффект составил 15%. По [S042]: эффект не превысил 8%.
```

Помечать `<!-- VERIFY: краткое описание -->` для проверки на этапе верификации.

#### Чего НЕ должно быть в финальном тексте

- Оригинальные ID агентов — есть в справочниках
- Голые URL
- Ссылки `[Cxx]` без Short-названия
- Дублирование источников, входящих в кейс

### 6.5. Этапы синтеза

#### 6.5.1. Синтез карточек

**Цель:** объединить данные по каждому кейсу из всех WRK-файлов в одну синтезированную карточку.

**Входные данные:**
- `_case_references.json` — справочник кейсов с маппингом агентов
- `_sources_references.json` — справочник источников
- WRK-файлы с нормализованными карточками

**Алгоритм (по одному кейсу за раз):**

1. Взять кейс из справочника (C01, C02, ...)
2. По маппингу `agents` найти оригинальные карточки в WRK-файлах
3. Извлечь карточки: `grep -A 50 "^### C03" WRK_Claude.md` и т.д.
4. Объединить данные:
   - Базовая информация — взять наиболее полную версию
   - Классификация — сверить теги, при расхождении зафиксировать
   - Описание — синтезировать, убрать дубли
   - Результаты/метрики — собрать все уникальные
5. Переатрибутировать источники:
   - Взять оригинальный S-ID из карточки агента (например, S025 из WRK_OpenAI_DeepResearch.md)
   - Найти в _sources_references.json запись, где `agents.[AgentCode]` равен этому ID (например, `agents.OAI_DR: "S025"`)
   - Заменить оригинальный ID на ключ найденной записи — это финальный S-ID (например, S001)
6. Обновить `_case_references.json`:
   - Заполнить поле `sources` для этого кейса `"sources": ["S001", "S003", ...]` и т.п.
7. Записать карточку в SYNTHESIS.md (секция Casebook)
8. Удалить использованные карточки из WRK-файлов, добавив метку `<!-- REMOVED: [Cxx] → SYNTHESIS.md/Casebook -->`

**Формат карточки:** см. TASK.md, секция 1.7 "Шаблон карточки"

**Проверка после каждой карточки:**
- [ ] Все источники переатрибутированы (нет оригинальных ID агентов)
- [ ] `sources` в справочнике обновлён
- [ ] Карточки в WRK-файлах заменены на метки REMOVED

**Проверка после всех карточек:**
- [ ] Все кейсы из справочника обработаны
- [ ] Секции Casebook в WRK-файлах содержат только метки REMOVED

#### 6.5.2. Кросс-синтез

**Цель:** синтезировать паттерны, ответить на ключевые вопросы исследования, выявить и зафиксировать противоречия.

**Входные данные:**
- SYNTHESIS.md с заполненным Casebook
- `_case_references.json` с заполненными `sources`
- `_sources_references.json`
- WRK-файлы: секции Observed Patterns, уникальные секции (`<!-- UNIQUE -->`)

**Секции для заполнения:**
- Observed Patterns
- Negative Cases
- Ключевые вопросы исследования
- Metrics Framework (если есть R2/R3 кейсы)
- Противоречия и верификация

**Алгоритм:**

1. **Сбор материала из WRK-файлов:**
   - Извлечь Observed Patterns из всех агентов
   - Извлечь уникальные секции (помечены `<!-- UNIQUE -->`)
   - Список ключевых вопросов из TASK.md

2. **Валидация паттернов:**
   - Паттерн у 2+ агентов → принимаем
   - Паттерн у одного агента → проверяем по синтезированным карточкам в Casebook
   - Расхождение между агентами → фиксируем в секции "Противоречия"

3. **Синтез по секциям (по одной за раз):**

   **Observed Patterns:**
   - Группировать по осям из TASK.md (тип решения, регион, стадия...)
   - Атрибутировать: кейсы `Short [Cxx]`, источники `[Sxxx]` если не в кейсе
   - Добавить собственные инсайты, если выявлены при синтезе — пометить `[Inference]`
   - Удалить обработанные паттерны из WRK, добавив `<!-- REMOVED: Observed Patterns → SYNTHESIS.md/Observed Patterns -->`

   **Negative Cases:**
   - Собрать кейсы со статусом PAU/DIS из Casebook
   - Добавить паттерны провалов из Observed Patterns агентов
   - Выделить общие причины неудач, если прослеживаются — пометить `[Inference]`
   - Атрибутировать по правилам 6.4

   **Ключевые вопросы исследования:**
   - Каждый вопрос из TASK.md → отдельная подсекция
   - Прямой ответ с атрибуцией
   - Evidence level (R0-R3)
   - Дополнительные инсайты по вопросу, если есть — пометить `[Inference]`

   **Metrics Framework:**
   - Только если есть кейсы R2/R3
   - Группировать по типам метрик
   - Атрибутировать источники метрик

   **Противоречия и верификация:**
   - Собрать все зафиксированные противоречия
   - Добавить неочевидные расхождения, выявленные при синтезе
   - Методологические ограничения исследования

4. **Фиксация противоречий:**
   - Формат: "По Short [Cxx]: A. По [Sxxx]: B."
   - Все противоречия → `<!-- VERIFY: описание -->`

5. **Удаление из WRK:**
   - После обработки каждой секции удалять использованный материал
   - Метка: `<!-- REMOVED: [название] → SYNTHESIS.md/[секция] -->`

- [ ] Все секции из списка заполнены (или помечены N/A)
- [ ] Все паттерны атрибутированы по правилам 6.4
- [ ] Собственные инсайты помечены `[Inference]`
- [ ] Противоречия зафиксированы с `<!-- VERIFY -->`
- [ ] Секции Observed Patterns в WRK-файлах содержат только метки REMOVED
- [ ] Уникальные секции в WRK-файлах содержат только метки REMOVED

### 7.1. Верификация противоречий

**Цель:** разрешить все `<!-- VERIFY: ... -->` метки в SYNTHESIS.md.

**Алгоритм (по одному противоречию):**

1. Извлечь VERIFY-метку и контекст
2. Сформулировать поисковый запрос (учесть: синтез может быть на RU/другом языке, первичка — основная на EN)
3. Выполнить `web_search` по сути противоречия
4. Определить исход:

| Исход | Что случилось | Действие |
|-------|---------------|----------|
| **Разрешено в пользу одного** | Подтвердил A, опроверг B | Исправить текст, обновить источники (добавить/удалить) |
| **Оба правы** | Интерпретация, неполнота, про разное | Уточнить контекст в тексте, источники валидны |
| **Оба неправы** | Нашёл третью версию, оба источника ошибочны | Исправить текст по новым данным, добавить источник, пометить в отчете старые как ненадёжные |
| **Неразрешимо** | Реальное противоречие в домене, разные исследования дают разные результаты | Зафиксировать в тексте как "данные расходятся" с атрибуцией обоих |
| **Не проверяемо** | Пэйволл, сайт упал, не гуглится | Эскалация пользователю |

5. Обновить SYNTHESIS.md: удалить VERIFY-метку, исправить/уточнить текст
6. Обновить справочники если изменились источники:
   - `_sources_references.json` — добавить новый источник / пометить удалённый
   - `_case_references.json` → `sources` — обновить список источников кейса

## Фаза 8: Executive Summary и финализация

**Цель:** трансформировать SYNTHESIS.md в SUMMARY.md — сжатый документ для принятия решений целевой аудиторией.

### 8.1. Определение фокуса

Фокус определяется либо на основании дизайна исследования TASK.md, либо в диалоге с пользователем.

Перед началом работы зафиксировать:
- **Для кого** — кто читает (роль, уровень)
- **Назначение** — какое решение принимает на основе отчёта

Фокус определяет что выносить в топ, что сокращать, какие детали сохранять.

**Примеры фокусов:**

| Аудитория | Назначение | Акценты |
|-----------|------------|---------|
| PMO, продукт | Инсайты, валидация гипотез | Паттерны, механики, что работает/не работает |
| CTO, архитектор | Оценка решений | Технологии, интеграции, масштабирование |
| Инвестор | Due diligence | Метрики, unit economics, риски |
| Исследователь | Обзор домена | Методология, gaps, направления |

### 8.2. Каркас SUMMARY.md

```markdown
# [Заголовок исследования]

Синтез N исследований.
Для кого: [аудитория]
Назначение: [решение, которое принимает читатель]
Дата: YYYY-MM-DD
Источники: <-- перечислить агентов по полным именам, не коды -->

## Цель исследования
[из SYNTHESIS, без изменений]

## Legend
[из SYNTHESIS, без изменений]

## Methodology
[сжато: подход + ключевые ограничения]

### Сравнение агентов
[таблица: кейсов, источников, P1%, качественная оценка]

## Топ-10 кейсов
[критерий отбора определяется фокусом]

## Топ-5 негативных кейсов
[что пошло не так, почему важно]

## Observed Patterns
[ключевые, с атрибуцией]

## Ключевые вопросы исследования
[ответы сжато]

## Metrics Framework
[если есть R2/R3 данные]

## Противоречия
[неразрешённые или критичные для решения]
```

### 8.3. Принципы трансформации

**Что переносить из SYNTHESIS:**
- Выводы и паттерны — сжато, с атрибуцией
- Методология — ключевое, без деталей по файлам
- Ответы на вопросы исследования — кратко

**Что убирать:**
- Полный Casebook — заменяется топ-таблицами
- Appendix Sources — атрибуция остаётся inline
- Служебные секции (справочники, маппинги)
- Детали нормализации

**Топ-кейсы и негативные кейсы:**

Не карточки, а таблицы. Один кейс = одна строка с ключевым результатом.

Формат топ-кейсов:
```markdown
| # | Кейс | Регион | Тип | R | Ключевой результат |
|---|------|--------|-----|---|-------------------|
| 1 | Short [Cxx] | ... | TAG | Rx | Что делает его топом + [Sxxx] |
```

Формат негативных кейсов:
```markdown
| # | Кейс | Регион | Причина провала | Урок |
|---|------|--------|-----------------|------|
| 1 | Short [Cxx] | ... | Что пошло не так | Вывод для читателя |
```

После таблицы негативных — паттерны провалов (3-5 пунктов, обобщение).

**Критерий отбора топ-кейсов:**
- Определяется фокусом исследования и аудиторией
- Указывать явно над таблицей: *"Критерий: R3 первые, затем R2 с наибольшим охватом"*

**Свобода агента:**
- Агент определяет какие паттерны ключевые для данной аудитории
- Агент может добавить секции, релевантные фокусу
- Агент может изменить порядок секций под логику чтения
- Агент добавляет инсайты, которые считает важными для читателя

### 8.4. Оформление

**Атрибуция:**
- Кейсы: `Short [Cxx]` — всегда с названием
- Источники: `[Sxxx]` — inline, без отдельного Appendix
- Inference: помечать `(Inference: N=X)` если N<3

**Язык:**
- Термины — расшифровка при первом использовании
- Если отчёт на RU/другом языке — минимум англицизмов, локальные эквиваленты

- [ ] Критерий отбора указан явно
